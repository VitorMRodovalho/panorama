## TL;DR (5 bullets)

1. **Three audit waves complete.** ~126 distinct findings documented, 61 filed as labelled GitHub issues, 3 tracking parents (#27 Wave 1, #54 Wave 2, #73 Wave 3). Full reports under [`docs/audits/`](./). Three audit commits in `main`: `d07251a`, `4aea39e`, `33ef07a`.
2. **Backend engineering is strong for a 6-day-old codebase.** ADR discipline, migration safety, RLS strategy, typed boundaries are all above average. **The gaps are concentrated in three places: the web UI (~10 % built), CI enforcement theatre (lint echo-stubbed, Trivy double-bypassed), and one dead runtime path (ADR-0016 maintenance module accepted and migrated but zero code).**
3. **Six issues are ship-blockers for any pilot tenant.** In order of biggest rock first: #74 (maintenance module, 3–4 weeks solo), #28 (OIDC email_verified takeover), #30 (RLS 0014 raw GUC), #75 (invitation UI), #31 (`lastReadMileage` never written), #29 (inspection module forbid violation).

5. **The 2026-04-20 MCP CVE family does NOT expose Panorama's product surface today** (zero `@modelcontextprotocol/*` in the tree, zero AI features in production). It surfaces three derived concerns now documented: contributor-workstation runbook ([`docs/runbooks/dev-environment-ai-tooling.md`](../runbooks/dev-environment-ai-tooling.md)), plugin-sdk redesign block (#84), and ADR-0017 AI governance draft ([`docs/adr/0017-ai-llm-integration-principles.md`](../adr/0017-ai-llm-integration-principles.md)).

## What was explicitly NOT audited

So the next auditor (human or agent) knows where scope ended.

- **Live runtime behaviour.** Every wave was static analysis + grep + read. Nothing was deployed or executed. `pnpm audit` and `pnpm typecheck` were run, nothing else. No actual EXPLAIN output from a populated DB; [`apps/core-api/prisma/perf/baseline.sql`](../../apps/core-api/prisma/perf/baseline.sql) was produced so the team can generate it, not run for you.
- **Production deployment surface.** Ingress WAF, TLS termination, CDN config, Supabase RLS at the managed layer, Fly.io / Cloudflare resource configuration — out of scope. SECURITY.md mentions them; the audit did not verify them.
- **Penetration testing.** Zero active probing. No attempted exploits of the supply-chain advisories, no actual account takeover attempt via the OIDC `email_verified` gap, no tenant-leakage probing. Everything is reasoning from source.
- **Performance under load.** Wave 2c predicted plans at scale; no load test was run.
- **OIDC end-to-end against a real IdP.** Issue #28 flags a code-level gap; the actual Google / Microsoft flow has not been exercised in staging. Wave 3a PILOT-10 covers this as pre-pilot smoke-test.
- **Docker image build / runtime security.** Trivy scans filesystem in CI; runtime config of the deployed container (seccomp, read-only fs, capabilities drop) was not audited.
- **Helm / Terraform / infra-as-code.** README references them; directories are empty / placeholders. Wave 1 PROD-13 flags the gap; no depth.
- **Mobile app.** `apps/mobile/` is empty. Wave 1 OPS-06 + ARCH-12 flag the gap; no depth.
- **Database migrations under concurrent traffic.** Wave 2c DATA-11 flagged missing `lock_timeout` / `statement_timeout` in migrations. Not stress-tested.
- **Upstream library source review.** Supply-chain triage (Wave 3b) reasoned from advisories + package code surface; did not line-by-line review upstream patch diffs for any of the 22 CVEs.

## How to run the next audit wave

When the MVP cut ships and you want to re-audit:

1. **Re-run the orientation pass.** Read [`docs/audits/2026-04-23-wave-1.md`](./2026-04-23-wave-1.md) §"What you're looking at" to see the shape of a full audit.
2. **Dispatch the six specialised subagents in parallel** against the new commit (security-reviewer, tech-lead, data-architect, ux-critic, persona-fleet-ops, product-lead). Each has a persona descriptor under `.claude/agents/`; the wave-1 prompts are embedded in this audit trail as reproducible templates.
3. **Diff findings against closed issues.** If any wave-1/wave-2/wave-3 issue was closed without a code change, verify the issue is genuinely resolved; don't let dismissals leak into regressions.
4. **Track new findings under `audit:wave-N`.** Keep the one-label-per-wave convention so audit trails stay separable.
5. **Commit the new wave's consolidated report** to `docs/audits/YYYY-MM-DD-wave-N.md`.
6. **Update this handoff doc.** It is the living entry point.
