# Audit Wave 1 — 2026-04-23

- **Commit audited:** `c0f5c2b` (main)
- **Repo state:** pre-alpha (bootstrapped 2026-04-17), 14 migrations, 16 ADRs, 32 e2e/integration tests, 0 unit tests
- **Auditors (parallel passes):** security-reviewer, tech-lead, data-architect, ux-critic, persona-fleet-ops, product-lead
- **Method:** full-repo QA/QC (not a diff review). Each reviewer received the same repo snapshot and a domain-scoped charter. Findings below are deduplicated and cross-referenced.
- **Registered as:** individual GitHub issues labelled `audit:wave-1` (one per finding) + a parent tracking issue. This doc is the durable record.

## TL;DR for the maintainer

Panorama's backend engineering is **exceptionally strong for a 6-day-old codebase** — ADR discipline, migration safety, RLS strategy, typed boundaries, CI plumbing. Wave 1 found **6 critical-severity issues** plus 15 high-severity, concentrated in three themes:

1. **Code-vs-ADR drift.** ADR-0016 maintenance flow is accepted and migration 0014 shipped, but `Asset.lastReadMileage` is never written, `panorama.reservation.checked_in_with_damage` is never emitted, and `MaintenanceTicketSubscriber` doesn't exist. The dominant path for auto-suggested maintenance tickets (70% of real-world tickets per ADR-0016's own analysis) is dead on arrival.
2. **Product-reality gap.** README/feature-matrix claim capabilities the web layer doesn't deliver. The web app has 12 pages, no navigation, no asset CRUD, no user management, no blackout UI, and ~80% of user-visible strings are hardcoded English despite a well-maintained trilingual i18n bundle. A PT-BR/ES user sees a jarring mixed-language surface.
3. **Silent CI gates.** Lint is echo-stubbed for core-api and web. Trivy SCA is double-bypassed (`continue-on-error: true` AND `exit-code: '0'`). The `ensure-community-complete` CI job is a fragile grep, not a functional test. CONTRIBUTING.md promises enforcement that doesn't exist.

Plus one **critical security gap** (SEC-01): OIDC login does not check `email_verified`, enabling account takeover via IdPs that allow unverified sign-in.

None of the critical issues blocks backend development velocity. All should be triaged before any pilot tenant touches the system.

## Proposed next waves (deeper investigations)

Wave 1 surfaced several areas where a single-pass audit is insufficient. Recommended follow-ups, in priority order:

1. **Wave 2 — RLS compliance sweep.** ADR-0015 reports 25 `runAsSuperAdmin` call sites. For each, determine: is super-admin actually needed, or is the service avoiding `runInTenant` boilerplate? Reservation module is the primary suspect (ARCH-02). Goal: reduce `runAsSuperAdmin` surface to the minimum justified set (audit writes, maintenance sweeps, auth membership lookups, notification dispatcher claims).

2. **Wave 2 — Notification bus end-to-end trace.** Walk every `enqueueWithin` call site. Verify (a) event type is registered in `NOTIFICATION_PAYLOAD_SCHEMAS`, (b) Zod schema matches the actual payload shape, (c) a `ChannelHandler` exists that `supports()` the event type. The missing `checked_in_with_damage` event (ARCH-15) suggests other dead-end or unregistered events may exist.

3. **Wave 2 — Query performance baseline.** Run `EXPLAIN (ANALYZE, BUFFERS)` on the 4 queries flagged by data-architect. Produce a baseline doc so later perf regressions have a reference point. Include the notification dispatcher `FOR UPDATE SKIP LOCKED` path under concurrent workers.

4. **Wave 2 — CI hardening pass.** Configure ESLint for core-api. Fix Trivy to actually block. Replace the grep-based community gate with an import-graph check. Add coverage reporting. Add an `e2e` job for the turbo `e2e` pipeline (currently configured but never run). Add a migration-convention checker (every migration dir has `migration.sql`, `ROLLBACK.md`, `rls.sql`).

5. **Wave 3 — Pilot readiness gap.** Given PROD-12 (web UI ~10% built) and ARCH-03 (maintenance flow dead-on-arrival), what is the minimum viable surface for a real pilot tenant to exercise the product? Define the must-work flows (book, approve, checkout with mileage, inspect, check-in with damage, flag maintenance, resolve, book again). Everything outside that scope is deferred.

6. **Wave 3 — Supply-chain / deps triage.** Per-advisory exploitability pass on the 22 `pnpm audit` findings. Categorize: direct vs transitive, reachable vs dead path, DoS vs RCE. Close the Dependabot queue or write explicit `.trivyignore` entries for accepted transitive DoS.
