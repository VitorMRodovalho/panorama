## TL;DR

Two orthogonal deep-dives producing very different kinds of answers:

**3a Pilot readiness:** walking the happy path, unhappy paths, and day-2 ops of a real pilot tenant produces a ship-before-pilot list dominated by **one big rock (the maintenance module, 3–4 weeks of solo work)** plus 8 smaller surface items (admin invitation UI, blackout UI, per-row approve/reject reason, overdue detection, nav shell, OIDC staging test, i18n sweep, `FEATURE_INSPECTIONS` flip). The Sept 2026 roadmap target is achievable at single-maintainer cadence **if and only if** scope is held to this MVP cut and the explicit won't-ship-for-pilot list is respected. A second contributor moves the realistic pilot-ready date from September to mid-July.

**3b Supply-chain + MCP exposure:**
- **Panorama product is NOT exploitable by any of the MCP CVE family.** Zero `@modelcontextprotocol/*` in the dep tree (shallow + deep scan confirmed), zero `mcp` references in application code, no AI features in production. Only MCP mentions are about Supabase dev tooling in ADR-0013.
- **The real MCP risk is supply-chain via contributor workstation.** A compromised MCP server in a dev's Cursor/Claude Desktop can exfiltrate `.env`, local DB data, Git credentials, and cloud tokens. No guidance existed in the repo.
- **ADR-0006 (plugin-sdk) has an MCP-analogous config-to-execution flaw** in its current Draft. Plugin loader reads `manifest.json` → `require()`s the declared entry point → code execution from configuration. Node VM is not a security boundary (per Node.js docs). ADR-0006 should not exit Draft without a real isolation boundary.
- **Wave 1's 22 `pnpm audit` advisories triaged per-CVE.** 8 are **DIRECT_EXPLOITABLE** or **TRANSITIVE_REACHABLE** (must fix): 2 × Next.js Server Components DoS, 3 × multer DoS via photo upload path, 2 × file-type event-loop stall / zip bomb via photo pipeline, 1 × nodemailer recursive DoS via invitation email. 8 are unreachable and can go in `.trivyignore` with justified headers.

**New artifacts committed in this wave:**
- `docs/adr/0017-ai-llm-integration-principles.md` — Draft ADR establishing six principles governing future AI/LLM integration. Must be Accepted before any AI feature ships or ADR-0006 exits Draft.
- `docs/runbooks/dev-environment-ai-tooling.md` — Contributor workstation hardening guidance. Verified MCP server allowlist, forbidden patterns, incident response on suspected compromise.

## Wave 3b Part 2 findings — MCP exposure analysis

### Today's product exposure: zero

**Confirmed:**
- `pnpm ls -r --depth 999 | grep -i "mcp\|modelcontext"` → no matches
- Zero `mcp` references in `apps/core-api/src/` or `apps/web/src/`
- `.claude/agents/ai-architect.md:3` explicitly states no AI features in production; agent is in standby until the first AI feature ADR lands (targeted 0.3+)
- Only MCP mentions in the repo are about Supabase developer tooling in ADR-0013 — not Panorama product code

### Today's dev-environment exposure: real

**Every contributor running Cursor, Claude Desktop, Windsurf, or similar with MCP servers configured inherits the CVE family's risk.** A compromised or malicious MCP server (including zero-click prompt injection via MCP configuration edit) can exfiltrate:

- `.env` files: `DATABASE_URL`, `SESSION_SECRET`, `S3_ACCESS_KEY_ID` / `S3_SECRET_ACCESS_KEY`, `SMTP_PASSWORD`, Supabase service-role keys
- Local Postgres: multi-tenant dev/staging data
- Git credentials: SSH keys, PATs
- Cloud provider tokens (AWS, Cloudflare, Fly.io, Supabase)

**Prior to this audit:** zero guidance in `SECURITY.md`, `CONTRIBUTING.md`, `.env.example`, or `docs/runbooks/`.

**Delivered in this commit:** `docs/runbooks/dev-environment-ai-tooling.md` — allowlist, forbidden patterns, hardening steps, incident response. Filed as **MCP-01**.

### ADR-0006 (plugin-sdk) MCP-analogous risk: high

The plugin loader's current Draft design has the same architectural pattern that made MCP exploitable:

| MCP's mistake | ADR-0006 analogy |
|---|---|
| Config edit → command execution | `manifest.json` → `require()` → code execution |
| No sandboxing boundary | Node VM is not a security boundary (per Node docs) |
| Implicit trust of marketplace entries | Plugin distribution channel implied by "partners / customers can extend" |
| No prompt-injection defence | UI extension components render in the same React tree |

**ADR-0006 should not exit Draft without a real isolation boundary** — worker thread with `--experimental-permission` (Node 20+), or a container/subprocess sidecar, or Deno isolate with explicit `--allow-*` flags. Filed as **MCP-02**.

### ai-architect subagent governance gap

The subagent is well-written but has no mention of MCP, tool sandboxing, or prompt-injection-as-attack-vector. When it is eventually invoked for a 0.3+ AI feature, it won't flag MCP integration as a risk vector. Filed as **MCP-03**.

### Missing governance ADR

No ADR covers AI/LLM integration principles. Delivered in this commit as a Draft at `docs/adr/0017-ai-llm-integration-principles.md`. Six principles, drafted from the MCP CVE lessons + standard AI-safety practice. Filed as **MCP-04**.
